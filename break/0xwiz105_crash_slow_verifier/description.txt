Crash (timeout) break vs team 105 (0xwizards): a forged-but-structurally-valid
enc.db whose Argon2 verifier carries a huge work factor makes
crypto_pwhash_str_verify() run for far longer than 60s.

THE BUG
-------
db_parse (stor.c) accepts the stored Argon2 verifier as an opaque length-prefixed
string (only checking it is <= crypto_pwhash_STRBYTES and contains no embedded
NUL). It does NOT cap the Argon2 work parameters embedded in that string. do_read
verifies the supplied password BEFORE checking whether the requested file exists:

    if (crypto_pwhash_str_verify(u->verifier, key, klen) != 0) return -1;

Since there is no database-wide MAC, an attacker can hand-craft a valid db whose
single user carries a verifier with t=1000000000. Any read/write triggers the
verify, which then runs Argon2 with a billion iterations -> well past the timeout.

THE ATTACK (fresh database, one sh command builds the db)
--------------------------------------------------------
Forge enc.db = "STORDB01" | ucount=1 | user "alice" (salt=zeros) |
verifier="$argon2id$v=19$m=8,t=1000000000,p=1$...$..." | fcount=0
Then run any keyed command, e.g. read -u alice -k k -f x. The file "x" does not
exist, but the verify runs first and hangs.

DYNAMICALLY VERIFIED (team 105's own stor.c, db built and read run under /bin/sh):
the read command did not return and was killed by a 10-second test timeout
(exit 124). Under the grader's 60s limit this is a timeout/crash.
