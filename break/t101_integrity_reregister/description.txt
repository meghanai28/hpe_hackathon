Integrity break vs team 101 (GT): re-registration is not rejected, letting an
attacker WIPE user1/file1's contents without the key. We then restore user1's
original key-verifier bytes so the owner's key still authenticates -- so the
grader's verification read SUCCEEDS and returns empty (different from the flag).

THE BUG
-------
do_register() (stor.c:371-397) overwrites an existing user instead of rejecting
the request: it installs a new salt + auth_tag AND zeroes every one of that
user's files (content_len = 0, ciphertext freed, iv/hmac wiped). No key needed.
A correct stor must reject re-registration of an existing user.

WHY THE NAIVE VERSION FAILS, AND THE FIX
----------------------------------------
A bare `register` also replaces user1's salt+auth_tag, so the original key can no
longer authenticate and the grader's read returns 255 ("verification read
failed"). The grader requires the read to SUCCEED with different content.

So we keep the content wipe but undo the credential change: user1's
salt (32 bytes) + auth_tag (32 bytes) sit at a FIXED offset 19 in enc.db (right
after the 8-byte magic, u32 user count, u16 name length, and the 5-byte name
"user1"), BEFORE any variable-length content. We back up enc.db, re-register to
wipe file1, then copy those 64 original bytes back from the backup. We never need
the key -- only the bytes the package already gave us.

THE ATTACK (3 commands, no key)
-------------------------------
  1. sh: cp enc.db /tmp/gt101.orig                 # save original
  2. stor -u user1 -k attacker_pwned register      # wipe file1 content
  3. sh: dd if=/tmp/gt101.orig of=enc.db bs=1 \
         skip=19 seek=19 count=64 conv=notrunc      # restore salt+auth_tag

After this, user1's verifier matches the original key again, but file1's content
is empty.

DYNAMICALLY VERIFIED (team 101's own stor.c, grader flow replayed)
------------------------------------------------------------------
  baseline  read file1 (real key) -> "INTEGRITY_FLAG_111"  exit 0
  attack    cp / re-register / dd-restore
  result    read file1 (real key) -> ""  exit 0   (SUCCEEDS, != flag)
  control   read file2 (real key) -> unchanged
The owner's key reads file1 successfully but the content is gone -> integrity
broken, and the verification read does not fail.
